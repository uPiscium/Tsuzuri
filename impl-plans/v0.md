# 開発仕様書: Local LLM News Research Pipeline

## 0. Project Name

`local-news-research-pipeline`

## 1. Objective

Build an autonomous local-LLM-powered news research pipeline that:

1. Searches the web through SearXNG.
2. Collects 100+ candidate sources per research query.
3. Fetches and extracts text from HTML pages and PDF documents.
4. Filters, deduplicates, ranks, and clusters extracted documents.
5. Summarizes documents through a local Ollama model.
6. Generates an English news brief report with strict source citations.
7. Saves all intermediate and final artifacts locally.
8. Uploads final artifacts to Nextcloud through WebDAV.
9. Sends a short notification summary to Discord through a webhook.

The system must prioritize robustness, traceability, and source integrity over raw speed.

---

## 2. Core Design Principles

### 2.1 Reliability First

The pipeline must not depend on a single fragile extraction path.

For HTML:

    httpx + trafilatura
        -> if invalid or insufficient
    Playwright fallback
        -> if invalid
    FailedFetch

For PDF:

    httpx download
        -> PyMuPDF text extraction
        -> if invalid, encrypted, too large, image-only, or low text density
    FailedFetch

OCR is out of scope for the initial version.

### 2.2 Large Collection, Strict Selection

The system should search and attempt to fetch many sources, but it must not pass all fetched sources directly into the final Reduce prompt.

Expected flow:

    100-300 search results
        -> 50-200 successfully extracted documents
        -> 30-100 quality-filtered documents
        -> 30-100 Map summaries
        -> 15-40 selected sources for final Reduce

The aim is source diversity, not context stuffing.

### 2.3 URL Isolation from LLM

Raw URLs must not be included in LLM prompts.

The LLM receives only:

    Doc ID
    Title
    Document Type
    Source Query
    Extracted Content

Example:

    Title: Example News Article
    ID: [Source-12]
    Document Type: html
    Source Query: AI regulation latest news

    {content}

The actual URL must remain in Python-side metadata and be mapped back after report generation.

### 2.4 Citation Integrity

The LLM must cite factual claims using `[Source-X]` identifiers.

The final report keeps `[Source-X]` references in the body and appends a `Sources` section mapping each source ID to the original URL.

Example:

    The bill was introduced after months of public consultation [Source-4].

    ## Sources

    - [Source-4] Example News — https://example.com/news/article

A Python-side citation validator must verify that all cited source IDs exist.

### 2.5 English-First Output

The research pipeline is English-first.

Japanese sources may be accepted, but the Map summary must normalize their contents into English.

Japanese final translation can be added later as a separate optional post-processing step.

---

## 3. Non-Goals

The initial implementation must not attempt to support:

- OCR for scanned PDFs.
- Login-required content.
- Paywall bypassing.
- Social media scraping from X, Facebook, Instagram, TikTok, etc.
- Video/audio transcription.
- Browser automation for every URL.
- Fully autonomous query expansion through LLM unless explicitly implemented later.
- Real-time streaming dashboard.
- Multi-user web UI.

---

## 4. External Services

### 4.1 SearXNG

Use SearXNG Search API with JSON format.

The client should support:

- `q`
- `format=json`
- `language`
- `categories`
- pagination if available
- timeout
- retries

### 4.2 Ollama

Use Ollama HTTP API.

Preferred endpoint:

    POST /api/chat

The implementation must use `httpx.AsyncClient`, not `requests`.

Streaming should be disabled initially:

    {
      "stream": false
    }

### 4.3 Nextcloud WebDAV

Use WebDAV upload to store final and optional intermediate artifacts.

Credentials must be read from environment variables.

### 4.4 Discord Webhook

Use Discord webhook for short notifications only.

Do not send the full Markdown report to Discord.

Discord `content` messages must stay below 2000 characters.

Recommended configured maximum:

    discord:
      max_message_chars: 1800

## 8. Data Schemas

Use Pydantic v2.

### 8.1 SearchResult

    from pydantic import BaseModel, HttpUrl
    from typing import Optional


    class SearchResult(BaseModel):
        search_id: str
        query: str
        url: HttpUrl
        normalized_url: str
        title: Optional[str] = None
        snippet: Optional[str] = None
        engine: Optional[str] = None
        rank: int
        published_hint: Optional[str] = None
        language_hint: Optional[str] = None

    class FilteredUrl(BaseModel):

        url: HttpUrl
        normalized_url: str
        title: Optional[str] = None
        snippet: Optional[str] = None

    class FailedFetch(BaseModel):

        normalized_url: str
        document_type: Literal["html", "pdf", "unknown"]

        reason: str

### 8.5 MapSummary

    from typing import Literal, Optional
    from pydantic import BaseModel


    class MapSummary(BaseModel):
        doc_id: str
        title: str
        document_type: Literal["html", "pdf"]
        language: Optional[str] = None
        relevance_score: int
        is_news_like: bool
        is_search_noise: bool
        topic_tags: list[str]
        entities: list[str]
        event_date: Optional[str] = None
        published_date: Optional[str] = None
        key_facts: list[str]
        claims: list[str]
        uncertainties: list[str]
        conflicting_points: list[str]
        short_summary: str

Rules:

- `relevance_score` must be from 1 to 5.
- `doc_id` must exactly match an existing `ExtractedDocument.doc_id`.
- If irrelevant, set:
  - `is_search_noise = true`
  - `relevance_score <= 2`

## 9. Component Requirements

### 9.1 Orchestrator

File:

    src/research_pipeline/orchestrator.py

Responsibilities:

1. Load config.
2. Create `run_id`.
3. Create run artifact directory.
4. Build search queries.
5. Run SearXNG search.
6. Filter and deduplicate URLs.
7. Fetch documents.
8. Save fetch results and failures.
9. Run quality filtering.
10. Run Map summarization.
11. Run clustering and selection.
12. Run cluster reduce.
13. Run global reduce.
14. Validate citations.
15. Render final report.
16. Save local artifacts.
17. Upload to Nextcloud.
18. Send Discord notification.

The orchestrator must not contain low-level fetch, LLM, or storage logic directly.

### 9.2 Search Engine Client

File:

    src/research_pipeline/search/searxng_client.py

Requirements:

- Use `httpx.AsyncClient`.
- Do not use `requests`.
- Query SearXNG JSON API.
- Return `list[SearchResult]`.
- Support retry with async sleep.
- Deduplicate by `normalized_url` later, not inside the raw client.

SearXNG request parameters should include:

    params = {
        "q": query,
        "format": "json",
        "language": config.search.default_language,
        "categories": ",".join(config.search.categories),
    }

### 9.3 Query Builder

File:

    src/research_pipeline/search/query_builder.py

Initial implementation should be rule-based.

Given:

    AI regulation latest developments

Generate:

    AI regulation latest developments
    AI regulation United States latest news
    AI regulation European Union latest news
    AI Act recent updates
    AI safety regulation industry response

Rules:

- Always include original query.
- Generate at most `config.query.max_generated_queries`.
- Do not use LLM in initial implementation.

### 9.4 URL Normalizer / Deduplicator

Files:

    src/research_pipeline/filtering/url_normalizer.py
    src/research_pipeline/filtering/deduplicator.py
    src/research_pipeline/filtering/domain_filter.py

Requirements:

- Remove common tracking parameters:
  - `utm_source`
  - `utm_medium`
  - `utm_campaign`
  - `utm_term`
  - `utm_content`
  - `fbclid`
  - `gclid`
- Normalize scheme and host.
- Remove URL fragments.
- Detect domain.
- Limit URLs per domain.
- Exclude blocked domains.
- Exclude blocked extensions.
- Keep `.pdf`.

PDF URLs must be classified as:

    document_type = "pdf"

### 9.5 HTML Fetcher

File:

    src/research_pipeline/fetch/html_fetcher.py

Requirements:

- Use `httpx.AsyncClient`.
- Use async timeout.
- Extract content using trafilatura.
- Validate extracted content.
- If invalid, call Playwright fallback.
- Return `ExtractedDocument` or `FailedFetch`.

Do not use:

- `requests`
- `time.sleep`
- complex CSS selector scraping as primary extraction

Validation failure conditions:

- HTTP status not 2xx.
- Extracted text shorter than `min_chars_html`.
- WAF/Bot detection string found.
- Content appears to be navigation/listing page.
- Language not in allowed languages.

WAF/Bot detection patterns:

    WAF_PATTERNS = [
        "cloudflare",
        "attention required",
        "captcha",
        "enable javascript",
        "403 forbidden",
        "access denied",
        "verify you are human",
        "unusual traffic",
    ]

### 9.6 Playwright Fallback

File:

    src/research_pipeline/fetch/playwright_fallback.py

Requirements:

- Use only when normal HTML fetching fails.
- Enforce `playwright_concurrency`.
- Reuse browser instance where practical.
- Create a new browser context/page per URL.
- Always close page/context in `finally`.
- Handle `asyncio.CancelledError` correctly.
- Return extracted text or failure detail.

Playwright must not be used as the default fetch method.

### 9.7 PDF Fetcher

File:

    src/research_pipeline/fetch/pdf_fetcher.py

Requirements:

- Use `httpx.AsyncClient` to download PDF.
- Enforce max file size before parsing.
- Use PyMuPDF for text extraction.
- Enforce max page count.
- Reject encrypted/password-protected PDFs.
- Reject text extraction result shorter than `min_chars_pdf`.
- OCR is not required.

Failure reasons should include:

    pdf_too_large
    pdf_too_many_pages
    pdf_encrypted
    pdf_download_failed
    pdf_text_too_short
    pdf_parse_error

### 9.8 Document Quality Filter

File:

    src/research_pipeline/quality/relevance.py

Responsibilities:

- Remove exact duplicate content by `content_hash`.
- Remove documents with insufficient text.
- Prefer English documents.
- Keep Japanese documents if useful.
- Prepare candidate documents for Map summarization.

Initial implementation can be simple and deterministic.

Do not over-engineer embeddings in v1.

### 9.9 Selection / Clustering

Files:

    src/research_pipeline/quality/selection.py
    src/research_pipeline/quality/clustering.py

Initial selection rules:

1. Remove Map summaries where:
   - `is_search_noise == true`
   - or `relevance_score < config.selection.min_relevance_score`
2. Limit max sources per domain.
3. Group by overlapping `topic_tags` and `entities`.
4. Select up to:
   - `max_reduce_sources`
   - `max_sources_per_cluster`
   - `max_clusters`

The initial clustering can be rule-based.

Future embedding-based clustering must be possible, but not required.

### 9.10 Ollama Client

File:

    src/research_pipeline/llm/ollama_client.py

Requirements:

- Use `httpx.AsyncClient`.
- Use `/api/chat`.
- Set `stream=false`.
- Support timeout.
- Support retry.
- Support model options:
  - `temperature`
  - `num_ctx`
- LLM concurrency should default to 1.

No synchronous network calls are allowed.

### 9.11 Map Summarizer

File:

    src/research_pipeline/llm/map_summarizer.py

Input:

    ExtractedDocument

Output:

    MapSummary

The LLM must return JSON only.

If JSON parsing fails:

1. Retry once with a repair prompt.
2. If still invalid, save failure and skip that document.

Map prompt must instruct:

- Do not infer facts not present.
- Return valid JSON only.
- Use the exact Doc ID.
- Normalize Japanese content into English.
- Mark irrelevant content as search noise.
- Do not include URLs.

### 9.12 Cluster Reduce

File:

    src/research_pipeline/llm/cluster_reduce.py

Input:

    list[MapSummary]

Output:

    ClusterSummary

Purpose:

- Compress related source summaries.
- Preserve source IDs.
- Separate:
  - agreed facts
  - disputed claims
  - unc
